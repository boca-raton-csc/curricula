Welcome to the Web—HTML

Concrete

HTML is a lot like concrete. You need it, and you need it to be stable.

Why HTML?

HTML was created for primarily writers. The idea is you can write a couple of paragraphs of text and easily mark what you want to be in italics or be bold.

Fast forward a few years it represents entire data structures and complex page layouts.

HTML is Made up of Tags

This is an HTML tag. HTML code is made up of many tags.

<div></div>

A Tag of Many Names

Tags can have any name. While it is possible to assign any name, not all names are standardized.

<div></div>
<p></p>
<h1></h1>
<strong></strong>

<george></george>

No Craziness

It is important you only use tags with a standardized semantic meaning. Here is a graphic with most of the ones currently in use today.

Need More Information!

Often there are more things you will need to describe an element. So we use a set of key value pairs, we call these attributes.

A word, followed by another phrase in quotes represents one key and it's value. Not all attributes do not require a value though as you can see with required.

<input type="password" name="verify-password" required>

An Identification Problem

If we can only use a certain set of elements, what about when we need to define something more specific? The answer is classes.

>div< is the most generic block level element (>span< is the most generic inline element). So when we need something specific we will start by using this element.

<div class="user">
  <h1 class="user--name">Sara Smith</h1>
  <div class="user--joined">Joined 3 months ago</div>
</div>

More Then One

You can apply multiple classes to describe things even more specifically.

<div class="size-large color-red shape-circle">Hello World</div>

Does it Make a Difference?

In vanilla HTML, classes do not change anything. You use classes so that your CSS and JavaScript can find these elements and add your custom styles/functionality.

Some Pretty Awesome Advanced Things

One thing you should remember whenever writing HTML is that you are writing things for the computer to understand.

Semantic data describes HTML that is so well written that a computer can easily interpret what the author meant. For example:

Jane Doe
<img src="janedoe.jpg" alt="Photo of Jane Joe"/>
Professor
20341 Whitworth Institute
405 Whitworth
Seattle WA 98052
(425) 123-4567
<a href="mailto:jane-doe@xyz.edu">jane-doe@illinois.edu</a>
Jane's home page:
<a href="http://www.janedoe.com">janedoe.com</a>
Graduate students:
<a href="http://www.xyz.edu/students/alicejones.html">Alice Jones</a>
<a href="http://www.xyz.edu/students/bobsmith.html">Bob Smith</a>

How is the computer supposed to know what any of that text actually means?

Much better :)

Thanks to a little (ok a lot) of HTML the computer can now understand what this persons name is, or her address, or her job etc.

<div vocab="http://schema.org/" typeof="Person">
  <span property="name">Jane Doe</span>
  <img src="janedoe.jpg" property="image" alt="Photo of Jane Joe"/>
  <span property="jobTitle">Professor</span>
  <div property="address"  typeof="PostalAddress">
    <span property="streetAddress">
      20341 Whitworth Institute
      405 N. Whitworth
    </span>
    <span property="addressLocality">Seattle</span>,
    <span property="addressRegion">WA</span>
    <span property="postalCode">98052</span>
  </div>
  <span property="telephone">(425) 123-4567</span>
  <a href="mailto:jane-doe@xyz.edu" property="email">
    jane-doe@xyz.edu</a>
  Jane's home page:
  <a href="http://www.janedoe.com" property="url">janedoe.com</a>
  Graduate students:
  <a href="http://www.xyz.edu/students/alicejones.html" property="colleague">
    Alice Jones</a>
  <a href="http://www.xyz.edu/students/bobsmith.html" property="colleague">
    Bob Smith</a>
</div>
